1. 概述
============================================================

四段式 AI 视频生成管道，输入可以是视频/图片/文档/文本，产出拼接完成的视频文件。

管道结构:

  用户输入
    │
    ▼
  阶段1: 获取分镜头脚本 → ⏸ 人工确认
    │
    ▼ storyboard.json
  阶段2: 获取参考图 → ⏸ 人工审核
    │
    ▼ ref_frames/
  阶段3: 生成视频（内部重试 + Grok 兜底）
    │
    ▼ 全部生成完毕
  阶段4: ffmpeg 拼接 → 最终视频

每一阶段结束时都有明确的分割点和人工干预入口。
用户可通过关键词跳过人工审核，实现全自动执行。


2. 核心数据格式
============================================================

2.1 storyboard.json
-------------------
阶段1 产出，阶段2/3 消费的统一分镜头数据格式。

{
  "overview": {
    "title": "视频标题",
    "script": "全局叙事结构（各场景逻辑关系、情绪递进）",
    "style": "全局视觉风格",
    "color_palette": ["主色1", "主色2"],
    "overall_rhythm": "整体节奏描述"
  },
  "shots": [
    {
      "id": 1,
      "timecode": "00:00:00-00:03:05",
      "duration_sec": 3.2,
      "shot_type": "景别",
      "composition": "构图方式（含元素位置）",
      "colors": "色调光影（光源方向+光照情况+明暗对比）",
      "content": "中文画面详细描述",
      "dynamics": "动态元素描述",
      "vfx": "视觉特效（粒子/光效/烟雾等）",
      "camera_movement": "运镜方式",
      "transition": "转场描述（最后一个为空）",
      "image_prompt": "Claude生成的图片提示词（侧重静态）",
      "video_prompt": "Claude生成的视频提示词（侧重动态）"
    }
  ],
  "meta": {
    "total_duration_sec": 44.3,
    "total_shots": 11,
    "source_type": "video|images|storyboard_doc|text_prompt",
    "has_reference_frames": false
  }
}

字段说明:
  - script: 新增全局字段，描述完整叙事结构
  - vfx: 新增字段，视觉特效描述
  - image_prompt: 侧重静态画面（主要参考 content, composition, colors, shot_type）
  - video_prompt: 侧重动态表现（主要参考 content, composition, dynamics,
    camera_movement, transition, vfx）
  - vfx 主要归 video_prompt（特效本质上是动态的）
  - 两者有重叠是正常的，非机械拼接，AI 参与生成

2.2 raw_analysis.json
---------------------
千问 VL 的原始输出，仅含 overview + shots 的描述字段（不含 image_prompt/video_prompt）。
Claude 在本地读取此文件，AI 参与生成 image_prompt 和 video_prompt。

2.3 ref_frames/
---------------
每个镜头一张参考图，命名 shot_01.jpg, shot_02.jpg ... shot_NN.jpg


3. 阶段1: 获取分镜头脚本
============================================================

目标: 无论输入是什么，产出经人工确认的定稿 storyboard.json。

3.1 四条输入路径
----------------

  1A: 视频输入
      1A.1 ffprobe 提取 duration, size, width, height, fps
      1A.2 智能压缩:
            - 短边 < 720px 且 ≤ 300MB → 直接发送
            - 短边 ≥ 720px → 压缩到 720p
            - 文件 > 300MB → crf 23 压缩，仍超则 crf 26 兜底
      1A.3 发送千问 VL 分析 (qwen3-vl-plus)
      1A.4 分析提示词 → 千问返回描述字段
      1A.5 JSON 修复 + 帧级时间码解析 → raw_analysis.json
      1A.6 覆盖率验证:
            - coverage ≥ 95% → 通过
            - coverage < 95% → while 循环分段（从 coverage-2s 裁剪继续分析）
      1A.7 Claude 本地生成 storyboard.json（六步）:
            Step 1: 读取 overview 全局约束
            Step 2: 去水印/Logo/字幕过滤（排除含触发词的子句）
            Step 3: 生成 image_prompt（侧重静态）
            Step 4: 生成 video_prompt（侧重动态）
            Step 5: 施加 script 全局影响（开场→暗调缓慢等）
            Step 6: 组装 storyboard.json（不含 grouping）

  1B: 图片输入
      判断是否有提示词模板:
        - B1 有模板 → 千问逐张填充描述
        - B2 无模板 → 询问用户选择
      千问输出后，Claude 按 1A.7 逻辑生成 image_prompt/video_prompt

  1C: 文档输入
      C1 文本脚本 (序号｜画面｜运镜) → 解析分隔符
      C2 Excel (.xlsx) → openpyxl
      C3 Word (.docx) → python-docx

  1D: 纯文本创意
      分析意图 → 提取主题/风格/时长 → 展开为镜头列表 → 构造 storyboard.json

3.2 阶段1 出口: 人工审核
-------------------------
输出分镜头摘要表格:

  | # | 时间码 | 时长 | 内容简述 | vfx | transition | image_prompt | video_prompt |

审核要点:
  - image_prompt 侧重静态画面描述
  - video_prompt 侧重动态/运镜/特效/转场
  - 两者有部分重叠正常，检查侧重点是否正确

用户可: 通过 / 修改 / 取消（只保留脚本不继续）

跳过审核触发词 (9个):
  直接做完 | 不用看脚本 | 不用审核 | 跳过审核 | 跳过人工 |
  跳过脚本审核 | 直接继续 | 全部做完 | 自动继续


4. 阶段2: 获取参考图
============================================================

目标: 基于定稿 storyboard.json，为每个镜头产出一张参考图。

4.1 跳过阶段2
--------------
若用户初始指令包含 (9个触发词)，跳过整个阶段2，1→3:
  文生视频 | 不要参考图 | 不用生图 | 不用参考图 | 不用截图 |
  纯文字生成 | text2video | text to video | 不要图

4.2 三条获取路径
----------------

  2A: 原始是视频 → 截帧 + 默认去水印
      2A.1 截帧: start_time + 10帧 (0.417s @ 24fps)
      2A.2 默认去水印: 统一中文提示词
           "保持原图画面内容完全不变，去除图片中的水印、logo、字幕，画面清晰干净"
           参数: 按原图比例, 2K, dreamina image2image, 每镜头 1 张
           仅用户明确说"不去水印"才跳过

  2B: 原始是图片 → 直接作为参考帧（默认不去水印）
      仅用户明确说"去水印"时才执行，逻辑同 2A.2

  2C: 原始是文档/文本 → 文生图（默认执行）
      用 shot.image_prompt 调用 dreamina text2image
      参数: 16:9, 2K, seedance2.0, 每镜头 1 张
      仅用户说"不用生图"才跳过

4.3 阶段2 出口
--------------

  第零步: 生图任务完整性检测
    - 循环检测所有 dreamina 生图任务，失败自动重试
    - 不设上限，直到全部成功

  第一步: 格式校验
    校验项: storyboard.json 存在? shots 非空? 每个 shot 有 prompt?
            ref_frames/ 存在? 数量与 shots 一致?
    - 不通过 → 无论何时都暂停，报告缺失项
    - 通过 → 有跳过指令时不报告，无指令时输出校验报告

  第二步: 人工审核图片质量
    - 展示参考图列表
    - 出口逻辑同阶段1
    跳过审核触发词 (8个):
      直接做完 | 不用审核 | 跳过审核 | 跳过人工 |
      不用看 | 直接继续 | 全部做完 | 自动继续


5. 阶段3: 生成视频
============================================================

目标: 逐镜头生成视频片段，含内部重试和 Grok 兜底。

5.1 默认流程（无明确指令时）
-----------------------------

  | 项目       | 默认值                       |
  |------------|-----------------------------|
  | 后端       | Dreamina Web（浏览器操控）     |
  | 生成方式   | 有参考图→image2video / 无→text2video |
  | 单位       | 逐镜头，时长=shot.duration_sec |
  | 画幅       | 16:9                         |
  | 分辨率     | 720p                         |
  | 模型       | seedance2.0                  |

5.2 积分阈值检查
-----------------
  总积分 = Σ(每个镜头的积分)
  估算: text2video ~12-24分, image2video ~25-50分（依时长而定）

  ≤ 1000 → 直接生成
  > 1000 → 暂停确认（用户说"不管积分"则跳过）

5.3 三种生成模式
-----------------

  模式 A: text2video
    无参考图，纯提示词。适用: 跳过阶段2 / 纯文本创意。

  模式 B: image2video（默认）
    有参考图 + 提示词。

  模式 C: multi image2video（仅分组时）
    触发: 用户明确说"按Xs分组"/"分组"。

5.4 后端参数统一
----------------
  | 参数       | Dreamina                           | Grok              |
  |------------|------------------------------------|-------------------|
  | 模型       | seedance2.0 / fast                 | —                 |
  | 画幅       | 1:1/3:4/16:9/4:3/9:16/21:9        | 2:3/3:2/1:1/9:16/16:9 |
  | 分辨率     | 480p/720p/1080p                    | 480p/720p         |
  | 时长       | 4s/6s/8s/10s/12s                   | 6s/10s            |
  | text2video | ✅                                  | ✅                |
  | image2video| ✅                                  | ✅                |

5.5 逐镜头重试 + 兜底链
------------------------

  for each shot:
    1. 提交 Dreamina → 监控 → 下载
    2. 瞬时错误（上传超时/网络）→ 同模型重试最多 3 次
    3. 非瞬时错误（审核拦截/内容策略）→ 换 Grok 兜底（不倒置提示词）
    4. Grok 也失败 → 标记 ❌

5.6 阶段3 出口
---------------
  全部成功 → 自动进入阶段4
  有失败项 → 暂停，提供 3 个选项:
    1. 不管失败项，直接用已成功的拼接
    2. 倒置提示词后用 Grok 再试 (调用 prompt-optimizer)
    3. 手动修改提示词后重新生成

7. 用户指令判断规则
============================================================

  1. 有明确跳过指令 → 严格按指令，不在中间询问
  2. 无明确指令 → 每个阶段结束后询问
  3. 例外（以下情况无论何时都暂停）:
     - 格式校验不通过（阶段2出口第一步）
     - 阶段3 积分阈值 > 1000（除非用户说"不管积分"）
     - 阶段3出口有失败项
  4. 跳过阶段2: 用户指令含文生视频相关关键词
